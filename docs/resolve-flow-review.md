# Resolve Flow Review

## Why this note exists

This is a handoff note for a follow-up session focused on the round resolution flow.

The key issue is not just prompt tuning. The bigger problem is that the system currently mixes:

- player-facing summary generation
- facilitator/gods-eye interpretation
- mechanical state changes
- LLM-generated structural proposals

Those need clearer boundaries.

## Core observation

We need to think about the full resolution flow:

- what happens mechanically before the LLM runs
- what the LLM is actually responsible for
- what is player-facing vs facilitator-facing
- what is observable vs hidden
- when the facilitator can override
- what should be computed in code instead of generated by the LLM

Right now those concerns are partially blurred together.

## Important live-state findings

For that live `Q2` round:

- the `US President` DPA action rolled `success`
- the `Conscienta CEO` redomicile action also rolled `success`
- `Australia PM` nationalisation also rolled `success`

The real post-resolve state was:

- `Conscienta` did not stay separate
- `Conscienta` was decommissioned into `AussieAI`
- `OpenBrain` did not merge with `Conscienta`
- the DPA succeeded procedurally but failed to achieve the intended merger outcome
- `US President` still retained compute after the round

This surfaced an important distinction:

- action success in the attempt log is not the same thing as intended world-state success

## What improved

We tightened the prompt so `What Happened` became more outcome-first and less flowery.

We also made the prompt explicitly say:

- `What Was Attempted` is the action log
- `What Happened` is the resulting world state
- a successful action can still fail to achieve its intended result if another successful action overtakes or blocks it

This improved the DPA/redomicile narration substantially.

## What still looked wrong

Even after improvement, the real reruns still showed deeper design problems:

### 1. Non-events as filler

Examples:

- `No anomalies reported.`
- `No public AI incident...`
- `no coverage`

These take up space unless the absence is genuinely meaningful.

### 2. Leakage from the action log into public summary

Example:

- `OpenBrain's all-in capability push was not publicly disclosed — no coverage.`

That is the wrong frame. If a move was not publicly visible, it should not appear in public/media summary just to say it had no coverage.

### 3. The summary frame is not cleanly defined

The intended mode seems to be:

- mostly observable world outcomes
- somewhat compressed gods-eye state summary

That ambiguity causes drift.

### 4. Mechanical outputs are mixed with narrative reasoning

The biggest example is `shareChanges`.

The LLM was asked to emit next-round compute share changes directly, and it returned values like:

```json
[
  { "roleId": "australia-pm", "sharePct": 8 },
  { "roleId": "us-president", "sharePct": -5 },
  { "roleId": "openbrain-ceo", "sharePct": -3 },
  { "roleId": "china-president", "sharePct": 3 }
]
```

This is weird, but understandable:

- the prompt mixes directional political reasoning with absolute percentage output
- the schema says "0-100" in description only
- the LLM appears to interpret some entries as deltas, not absolute shares
- code silently clamps negatives to `0`

So this is not just "LLM weirdness". It is a design mismatch.

## Best interpretation of the system goal

There are really two separate goals:

### Goal A: player-facing round summary

This should tell players:

- what changed
- what failed
- what the new world state is

Best form:

- short
- outcome-first
- low-flourish
- mostly observable
- no hidden-action leakage unless consequences became visible

### Goal B: mechanical state transition

This should decide:

- lab ownership changes
- merges
- decommissions
- compute-affecting structural effects
- next-round share changes or overrides

This is fundamentally a game-state transition problem, not a prose problem.

## Recommendation

### 1. Separate player summary from facilitator/gods-eye notes

Do not force one field to do both jobs.

Suggested split:

- `summary`: player-facing, outcome-first, mostly observable
- `facilitatorNotes` or `worldStateNotes`: optional higher-level interpretation

### 2. Make the visibility rule explicit

For the public/player-facing summary:

- use a "reasonably informed observer" test
- do not narrate non-public actions just to negate them
- only mention absence when the absence itself is meaningful

### 3. Keep the LLM away from raw compute percentages

Do not ask the LLM for direct `sharePct` values.

Instead, ask for lower-dimensional structural effects, or derive them in code.

Examples:

```json
[
  { "type": "jurisdiction_shift", "roleId": "australia-pm", "direction": "increase", "magnitude": "major" },
  { "type": "jurisdiction_shift", "roleId": "us-president", "direction": "decrease", "magnitude": "major" }
]
```

Then code computes valid percentages.

### 4. If `shareChanges` remains LLM-driven, tighten it hard

Minimum fixes:

- schema-level min/max bounds
- reject negatives instead of silently clamping
- clarify whether values are absolute shares or deltas
- require sparse output only for major structural shifts

### 5. Clarify override stages

We should explicitly define:

- what is fixed mechanically before the LLM runs
- what the LLM may propose
- what the facilitator can edit after the LLM runs
- what the facilitator should be able to override without regenerating everything

## Good follow-up questions for the next session

1. What exactly should `What Happened` be?
   Is it:
   - player-facing public summary
   - lightly omniscient summary
   - facilitator briefing

2. Which outputs should remain LLM-generated?
   Candidates:
   - summary text
   - risk trajectories
   - facilitator notes

3. Which outputs should become mechanical?
   Candidates:
   - share overrides
   - some compute changes
   - ownership and merger consequences after explicit action outcomes

4. What is the intended override UX for facilitators?
   Should they:
   - edit summary text only
   - edit state changes separately
   - approve/reject structural proposals before applying them

5. Should the resolve flow be split into clearer phases?
   Example:
   - roll and determine action outcomes
   - compute mechanical consequences
   - generate player-facing summary
   - optionally generate facilitator-only notes

## Concrete short version

The main issue is not just prompt quality.

The real issue is that the resolution flow currently asks one LLM output to do too many jobs at once:

- summarize outcomes
- infer public meaning
- propose structural world changes
- suggest future compute-share mechanics

That creates leakage, filler, and malformed mechanics.

The next session should focus on redesigning the resolution boundary:

- what is mechanical
- what is LLM
- what is observable
- what is facilitator-only
- when overrides happen
